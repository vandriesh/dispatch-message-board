# Architecture

Code organization follows the project's `react-gradual-architecture` skill
(`.agents/skills/react-gradual-architecture/SKILL.md`). The rules below are the subset that
binds this project. Read the skill for the full set.

## Modular by feature

The app is organized **by feature, not by file type**. There is no top-level `components/`,
`hooks/`, or `utils/` bucket — those are the folders everything ends up in when nothing has
an owner.

```
app/                  # Next.js router surface ONLY — see below
features/
  messages/           # the feed, composer, inline edit/delete
  auth/               # login, session
  <feature>/
    README.md         # what this feature is, its structure and specs
packages/
  ui-kit/             # @dmb/ui-kit — design-system primitives ONLY — see below
```

Features live at the repo root, beside `app/` rather than inside it, so the router surface
stays pure — nothing under `app/` exists that isn't a route. (The skill permits
`app/features/<feature>/`; this is the same intent, adapted to the App Router.)

Keep related files close to the feature (`organize-locality`). A feature owns its components,
its domain logic, and its types.

### The one exception: `@dmb/ui-kit`

Primitives live in a **workspace package**, `packages/ui-kit/src/components/`, imported as
`@dmb/ui-kit`:

```ts
import { Button, Input, Badge } from "@dmb/ui-kit";
```

This is the single carve-out from the no-`components/`-bucket rule, and it doesn't violate it,
because the rule is about *ownership*: a type-named bucket is bad when it holds domain code
that should have belonged to a feature. The ui-kit holds **design-system primitives** — Button,
Input, Select, Popover — that belong to no feature precisely because every feature uses them.

A package rather than a folder, because the boundary is then enforced by the module graph
rather than by discipline: `@dmb/ui-kit` cannot import from `features/`, so a primitive
*cannot* quietly grow domain knowledge.

The boundary is strict:

- **Allowed in the ui-kit:** files generated by `shadcn add`, and our restyling of them.
- **Never in the ui-kit:** anything that knows what a Message, a Tag, or a User is. A
  `MessageCard` is domain code — it lives in `features/messages/`, and it *uses* `Badge` and
  `Button` from the kit. If you need to import a domain type into the ui-kit, the component is
  in the wrong package.

Adding a primitive: run `shadcn add` **from `packages/ui-kit/`** (it has its own
`components.json`), restyle it to the tokens, then export it from `src/index.ts`. Do not create
a top-level `components/` for anything. No `components/shared/`, no `components/common/`.

## Next.js code follows Next.js conventions; domain code does not

Two different worlds, and the split is deliberate:

- **Anything Next.js-shaped follows Next.js naming and placement, exactly.** `page.tsx`,
  `layout.tsx`, `route.ts`, `loading.tsx`, `proxy.ts` — in the locations the framework
  dictates, spelled the way the framework spells them. Do not invent alternatives here.
  Remember this is Next.js 16: check `node_modules/next/dist/docs/` before assuming a
  convention (`middleware.ts` is now `proxy.ts`).
- **Everything else — the components that describe the domain, and the services themselves —
  lives under the feature package**, and stays framework-agnostic where it reasonably can.

Route files should stay **thin**: they wire, they don't implement. A `page.tsx` resolves
params and renders a feature component; a `route.ts` parses the request and calls a feature
service. The domain logic sits in `features/`, where it is testable without booting a router.

## Keep code together

Define types **where the data is born** — next to the function or component that produces or
owns them. **Never create a separate `types.ts`.** An interface that lives away from its data
goes stale, and a shared `types.ts` becomes a dumping ground that couples every feature to
every other.

The same instinct applies to interfaces between features: if a type is born in
`features/messages`, that is where it is exported from.

## No premature code splitting

**If it fits in ~100 lines, it stays in one file.** Do not pre-split a feature into
`Component` / `hooks` / `utils` / `types` before there is any pressure to.

Extract only when there is a *reason*, and let the reason name itself:

- **Scanning gets hard** — you're scrolling to understand one thing. Extract.
- **Concerns diverge** — UI, logic, and utils pulling in different directions. Split them.
- **Second use** — wait for the *second* consumer before hoisting anything into shared code.
  One use is not a pattern; it's a coincidence.

When extracting: hooks for React logic, plain functions for everything (helpers, dumb/presentation component).
